# AI 문서 판별 프레임워크

문서 이미지를 자동으로 분류하고 필드를 추출하는 AI 파이프라인 + 웹 서비스.

---

## 프로젝트 목표

업로드된 문서 이미지를 보고 **어떤 종류의 문서인지 판별**하고, **정해진 필드 값을 자동으로 추출**한다.

- 관리자: 모범 문서 이미지를 업로드 → AI가 필드 스키마를 자동 구성 → 저장
- 사용자: 문서 이미지 업로드 → 등록된 스키마 중 가장 유사한 것 매칭 → 필드 값 추출

---

## 개발 변천사

### 1단계 — VLM 블랙박스 접근 (초기 실험)

처음엔 **VLM(Vision Language Model)** 을 그대로 사용했다.

- 문서 이미지를 VLM에 통째로 넣어 필드 분류 및 내용 추출
- 결과: 성능이 좋았다. 필드 분류도 잘 되고 내용 추출도 정확했다

**한계 인식**

VLM은 문서뿐 아니라 모든 종류의 사진을 이해하는 범용 모델이다. 우리가 실제로 필요한 기능은 세 가지뿐이다.

| 필요한 것 | VLM 내부 담당 |
|-----------|--------------|
| 시각적 레이아웃 이해 | ViT (Vision Transformer) |
| 텍스트 추출 | 내장 OCR |
| 내용 이해 및 추론 | LLM 부분 |

→ 범용 시각 능력(ViT)이 불필요하게 포함된 블랙박스를 쓰는 대신, 각 기능을 분해해서 직접 구현하면 **우리 요구사항에 더 맞게 커스텀** 가능하다.

---

### 2단계 — 기능 분해 설계 (DiT + OCR + LLM)

VLM을 다음과 같이 분해했다.

```
ViT 제거
  ↓
DiT (Document Image Transformer) 로 대체
  → 문서 레이아웃 시각 임베딩 담당

OCR (PaddleOCR)
  → 텍스트 추출 담당

LLM (Qwen2.5-7B)
  → 스키마 구성 / 스키마 선택 / 필드 값 추출 담당
```

**DiT 선택 이유**: ViT는 자연 이미지 범용 모델이고, DiT(Microsoft)는 문서 이미지 특화 사전학습 모델이라 레이아웃 임베딩에 적합하다고 판단했다.

---

### 3단계 — DiT 한계 발견 및 구조 수정 (현재)

**문제**: Microsoft DiT는 사전학습 시 `편지 / 공문서 / 이메일 / 영수증` 같이 **레이아웃이 완전히 다른 대분류** 구분 능력으로 학습됐다.

우리가 구별해야 하는 문서들(소득증명서 vs 진료확인서 vs 처방전 등)은 레이아웃이 비슷한 **같은 카테고리 안의 세분류**라서, DiT 시각 임베딩만으로 스키마 벡터를 저장하면 유사도 구분이 거의 안 됐다.

**수정**: 시각 임베딩 단독 → **텍스트 임베딩으로 교체**

```
모범 문서 → OCR → 텍스트 → klue/roberta-base → 768dim 벡터 → DB 저장

사용자 문서 → OCR → 텍스트 → klue/roberta-base → 768dim 벡터
                                                        ↓
                                              코사인 유사도 → Top-3 후보
                                                        ↓
                                              LLM 최종 선택 + 필드 추출
```

## 현재 기술 스택

### AI 파이프라인

| 역할 | 모델 | 비고 |
|------|------|------|
| 텍스트 임베딩 | `klue/roberta-base` | 768-dim, mean pooling |
| OCR | PaddleOCR (Korean PP-OCRv4) | 회전 보정 + 전처리 포함 |
| LLM | `Qwen2.5-7B-Instruct` | 4-bit NF4 양자화, GPU 추론 |

### 인프라

| 구성 요소 | 기술 |
|-----------|------|
| 백엔드 API | FastAPI + Uvicorn |
| DB | SQLite (SQLAlchemy ORM) |
| 프론트엔드 | Next.js 16 + React 19 + Tailwind CSS |
| 프록시 | Next.js Route Handler (`/api/proxy/[...path]`) |
